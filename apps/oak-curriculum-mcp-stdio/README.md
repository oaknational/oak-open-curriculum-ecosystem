# Oak Curriculum MCP Server (STDIO)

MCP server that provides AI assistants with access to Oak National Academy's curriculum content.

**Architecture**: This server imports all MCP tool definitions from `@oaknational/curriculum-sdk`. The tools are generated at compile time from the OpenAPI schema - no manual tool definitions exist in this application. When the API changes, `pnpm type-gen` updates the SDK, and this server automatically has access to new/changed tools.

## Quick start

```bash
pnpm -C apps/oak-curriculum-mcp-stdio build
OAK_API_KEY=your_oak_api_key_here
```

Use with MCP clients that support STDIO (e.g. Claude Desktop, Cursor). See `.mcp.json` or `.cursor/mcp.json` for local configuration examples.

## Architecture

This application follows the standard structure:

- App wiring in `src/app/`
- Tools in `src/tools/`
- Integrations in `src/integrations/`

## Architecture Highlights

### Compile-time SDK Tool Generation

- The `@oaknational/curriculum-sdk` generates MCP tool metadata and validators at compile time from the OpenAPI schema.
- This server lists tools from the SDK's generated `MCP_TOOLS`, and delegates execution via `executeToolCall` with the injected client.
- **No runtime schema fetching and no manual mapping layers** - everything is pre-generated and fully typed.
- **Key benefit**: When the OpenAPI schema changes upstream, running `pnpm type-gen` automatically updates all tool definitions. This server immediately has access to new endpoints without any code changes.
- See [OpenAPI Pipeline Architecture](../../docs/architecture/openapi-pipeline.md) for the complete pattern.

## Tool surface

- Tools are generated from the SDK OpenAPI schema. Use Inspector or `tools/list` to discover available tools. The set matches the HTTP `/mcp` endpoint.
- Three additional search tools (`search-sdk`, `browse-curriculum`, `explore-topic`) are available when Elasticsearch credentials are configured. These provide semantic search across all four curriculum indexes (lessons, units, threads, sequences) plus typeahead suggestions and faceted browsing.

### Search tools (optional)

Set `ELASTICSEARCH_URL` and `ELASTICSEARCH_API_KEY` environment variables to enable the semantic search tools. Without these, the search tools return "not configured" errors; all other tools work normally.

## Behaviour and validation

- Implemented with MCP `McpServer`. Input validation uses SDK‑generated Zod shapes via `zodRawShapeFromToolInputJsonSchema`.
- Unknown tool and argument‑validation failures are surfaced as JSON‑RPC errors (thrown as `McpError` to clients).
- Execution or output‑validation failures return a single text content item containing a compact JSON error payload.

## How it works

- Uses `McpServer` directly; tools registered from SDK metadata.
- Validates tool arguments with Zod schemas generated by `@oaknational/curriculum-sdk` from the OpenAPI spec.
- Executes tools via `executeToolCall` on a client created from `OAK_API_KEY`.

## Request Tracing with Correlation IDs

Each tool execution is assigned a unique correlation ID for request tracing.

### How it works

- Correlation ID generated per tool invocation: `req_{timestamp}_{6-char-hex}`
- ID included in all log entries for that execution
- Logs written to file only (`.logs/oak-curriculum-mcp/server.log`)

### Using correlation IDs

**Filter logs by correlation ID:**

```bash
# Find all logs for a specific request
grep "req_1699123456789_a3f2c9" .logs/oak-curriculum-mcp/server.log

# See all correlation IDs (unique requests)
grep -o '"correlationId":"req_[^"]*"' .logs/oak-curriculum-mcp/server.log | sort -u
```

**Example log entry:**

```json
{
  "level": "INFO",
  "message": "Tool execution started",
  "correlationId": "req_1699123456789_a3f2c9",
  "toolName": "get-key-stages",
  "timestamp": "2025-11-06T12:00:00.000Z"
}
```

### Benefits

- Trace execution flow across async operations
- Debug errors by filtering logs
- Identify slow operations
- Correlate client requests with server logs

## Tool Execution Timing

All tool executions are automatically timed and logged with duration information. Slow operations (>5 seconds) are logged at WARN level for easy identification.

### How it works

- Every tool invocation starts a high-precision timer
- Duration is logged when execution completes
- Timing data includes both formatted ("1.23s") and precise (1234.56ms) values
- Slow operations are automatically flagged with `slowOperation: true`
- All timing logs include correlation IDs for request tracing

### Example logs

Tool executions are logged to `.logs/oak-curriculum-mcp/server.log` with timing information:

**Normal execution** (DEBUG level):

```json
{
  "level": "debug",
  "message": "Tool execution completed success",
  "toolName": "get-lessons-transcript",
  "correlationId": "req_1699123456789_a3f2c9",
  "duration": "234ms",
  "durationMs": 234.56
}
```

**Slow execution** (WARN level):

```json
{
  "level": "warn",
  "message": "Tool execution completed success",
  "toolName": "get-lessons-by-topic",
  "correlationId": "req_1699123456790_b4e3d0",
  "duration": "5.42s",
  "durationMs": 5420.12,
  "slowOperation": true
}
```

### Filtering and analyzing timing logs

**Find all slow operations:**

```bash
grep '"slowOperation":true' .logs/oak-curriculum-mcp/server.log
```

**Find all operations over 1 second:**

```bash
grep "Tool execution completed" .logs/oak-curriculum-mcp/server.log | \
  jq 'select(.durationMs > 1000)'
```

**Find slow operations for a specific tool:**

```bash
grep '"toolName":"get-lessons-by-topic"' .logs/oak-curriculum-mcp/server.log | \
  jq 'select(.durationMs > 1000)'
```

**Average execution time per tool:**

```bash
grep "Tool execution completed" .logs/oak-curriculum-mcp/server.log | \
  jq -s 'group_by(.toolName) | map({tool: .[0].toolName, avgMs: (map(.durationMs) | add/length)})'
```

**Trace a specific request by correlation ID:**

```bash
grep "req_1699123456789_a3f2c9" .logs/oak-curriculum-mcp/server.log
```

## Error Debugging

The stdio server enriches all tool execution errors with correlation IDs, timing information, and tool context for improved production debugging. Error logs are written to `.logs/oak-curriculum-mcp/server.log` (never to stdout, which is reserved for MCP protocol).

Error logs include:

- **Correlation ID**: Unique identifier for tracing tool executions
- **Timing**: Tool execution duration when the error occurred
- **Tool name**: The MCP tool that failed
- **Error details**: Original error message and execution/validation failure details

### Troubleshooting workflow for tool errors

1. **Identify failing tool** → Check MCP client error or tool name in logs
2. **Find recent errors** → `grep -i "error" .logs/oak-curriculum-mcp/server.log | tail -20`
3. **Get correlation ID** → Note the `correlationId` from the error log
4. **Trace execution** → `grep '<correlation-id>' .logs/oak-curriculum-mcp/server.log | jq .`
5. **Review timing** → Check if error correlates with slow operation warnings
6. **Analyze payload** → Examine tool input/output in the error payload
7. **Check validation** → For validation errors, review the schema requirements

### Log file location

All logs are written to: `.logs/oak-curriculum-mcp/server.log`

## Log File Management

The stdio server writes all logs to a file (never to stdout) to maintain MCP protocol integrity. Proper log file management ensures reliable debugging and prevents disk space issues.

### Default Configuration

- **Log Path**: `.logs/oak-curriculum-mcp/server.log` (relative to working directory)
- **Append Mode**: Logs append to existing file (not overwritten on restart)
- **Format**: Structured JSON (one log entry per line)
- **Location**: Configurable via `MCP_LOGGER_FILE_PATH` environment variable

### Log File Structure

Each line in the log file is a valid JSON object:

```json
{"level":"info","message":"Tool execution started","correlationId":"req_...","toolName":"searchLessons","timestamp":"2024-11-06T12:00:00.000Z"}
{"level":"debug","message":"Tool execution completed","correlationId":"req_...","duration":"234ms","durationMs":234.56}
{"level":"error","message":"Tool execution failed","correlationId":"req_...","context":{...}}
```

### Log Rotation Recommendations

The stdio server does not automatically rotate logs. Consider implementing log rotation to prevent unbounded disk usage:

**Using logrotate (Linux/macOS):**

Create `/etc/logrotate.d/oak-curriculum-mcp`:

```text
/path/to/.logs/oak-curriculum-mcp/server.log {
    daily
    rotate 7
    compress
    delaycompress
    missingok
    notifempty
    create 0644 user group
    postrotate
        # No need to restart; server appends to new file
    endscript
}
```

**Manual rotation:**

```bash
# Archive current log
mv .logs/oak-curriculum-mcp/server.log .logs/oak-curriculum-mcp/server.log.$(date +%Y%m%d)

# Compress old logs
gzip .logs/oak-curriculum-mcp/server.log.*

# Clean up old logs (older than 30 days)
find .logs/oak-curriculum-mcp/ -name "server.log.*.gz" -mtime +30 -delete
```

**Monitoring log file size:**

```bash
# Check current log file size
du -h .logs/oak-curriculum-mcp/server.log

# Monitor in real-time
watch -n 5 'du -h .logs/oak-curriculum-mcp/server.log'

# Alert if log exceeds 1GB
SIZE=$(du -m .logs/oak-curriculum-mcp/server.log | cut -f1)
if [ $SIZE -gt 1024 ]; then
    echo "WARNING: Log file exceeds 1GB"
fi
```

### Parsing Structured Logs

The server outputs JSON logs that can be parsed programmatically:

**Using jq (command-line JSON processor):**

```bash
# Pretty-print all logs
cat .logs/oak-curriculum-mcp/server.log | jq .

# Extract specific fields
cat .logs/oak-curriculum-mcp/server.log | jq '{level, message, correlationId}'

# Filter by log level
cat .logs/oak-curriculum-mcp/server.log | jq 'select(.level == "error")'

# Time range filtering
cat .logs/oak-curriculum-mcp/server.log | \
  jq 'select(.timestamp >= "2024-11-06T00:00:00Z" and .timestamp < "2024-11-07T00:00:00Z")'
```

**Using Node.js:**

```javascript
const fs = require('fs');
const readline = require('readline');

async function parseLogFile(filePath) {
  const fileStream = fs.createReadStream(filePath);
  const rl = readline.createInterface({
    input: fileStream,
    crlfDelay: Infinity,
  });

  const errors = [];
  for await (const line of rl) {
    try {
      const log = JSON.parse(line);
      if (log.level === 'error') {
        errors.push(log);
      }
    } catch (e) {
      console.error('Failed to parse log line:', line);
    }
  }

  return errors;
}

parseLogFile('.logs/oak-curriculum-mcp/server.log').then((errors) =>
  console.log(`Found ${errors.length} errors`),
);
```

**Using Python:**

```python
import json

def parse_log_file(file_path):
    errors = []
    with open(file_path, 'r') as f:
        for line in f:
            try:
                log = json.loads(line)
                if log.get('level') == 'error':
                    errors.append(log)
            except json.JSONDecodeError:
                print(f"Failed to parse log line: {line}")
    return errors

errors = parse_log_file('.logs/oak-curriculum-mcp/server.log')
print(f"Found {len(errors)} errors")
```

### Log Retention Recommendations

- **Active development**: Keep 7-14 days of logs
- **Production**: Keep 30-90 days depending on compliance requirements
- **Archive old logs**: Compress and move to long-term storage
- **Monitor disk usage**: Set up alerts if log directory exceeds threshold

### Ensuring Stdout Remains Clean

The stdio server MUST NOT log to stdout, as this will corrupt the MCP protocol stream. Verify stdout is clean:

**Check for stdout contamination:**

```bash
# Run server and capture stdout
node dist/bin/oak-curriculum-mcp.js > stdout.txt 2>&1

# Verify all stdout is valid JSON-RPC
cat stdout.txt | jq . > /dev/null
# If this succeeds, stdout is clean

# Look for non-JSON content
cat stdout.txt | grep -v '^{' | grep -v '^}'
# Should return nothing
```

**If you find stdout contamination:**

1. Check for `console.log()` calls in code (should use logger instead)
2. Verify logger is configured with file-only sink
3. Check dependencies for stdout writes
4. Review error handling (ensure errors don't write to stdout)

## Operational Debugging Workflows

Use these workflows to diagnose common issues with the stdio MCP server.

### Workflow 1: Investigating a Slow Tool Execution

**Scenario**: A tool execution takes longer than expected (>5 seconds).

**Steps:**

1. **Identify slow operations** in logs:

   ```bash
   grep '"slowOperation":true' .logs/oak-curriculum-mcp/server.log | tail -10
   ```

2. **Get correlation ID** from a slow operation log
3. **Trace the full execution**:

   ```bash
   grep 'req_1699123456789_a3f2c9' .logs/oak-curriculum-mcp/server.log | jq .
   ```

4. **Analyze timing breakdown**:
   - Check "Tool execution started" for start time
   - Check "Tool execution completed" for duration
   - Look for intermediate operations

5. **Compare with normal executions**:

   ```bash
   # Average timing for this tool
   grep '"toolName":"searchLessons"' .logs/oak-curriculum-mcp/server.log | \
     jq -s 'map(.durationMs) | add/length'
   ```

6. **Take action**:
   - If consistently slow: optimize the tool implementation
   - If intermittent: check for API rate limiting or network issues
   - If specific inputs: analyze input patterns

### Workflow 2: Investigating a Tool Execution Failure

**Scenario**: A tool execution fails with an error.

**Steps:**

1. **Find recent errors**:

   ```bash
   grep '"level":"error"' .logs/oak-curriculum-mcp/server.log | tail -5 | jq .
   ```

2. **Get correlation ID** from the error log
3. **Trace full execution**:

   ```bash
   grep '<correlation-id>' .logs/oak-curriculum-mcp/server.log | jq .
   ```

4. **Review error context**:

   ```json
   {
     "level": "error",
     "message": "Tool execution failed",
     "context": {
       "payload": "{\"toolName\":\"...\",\"toolExecutionError\":{...}}",
       "correlationId": "req_...",
       "duration": "145ms",
       "toolName": "searchLessons"
     }
   }
   ```

5. **Check common issues**:
   - **Input validation**: Check if tool arguments are valid
   - **API errors**: Review upstream API error messages
   - **Network issues**: Check timing for timeouts
   - **Authentication**: Verify API key is valid

6. **Take action**:
   - Fix invalid input parameters
   - Handle API errors gracefully
   - Increase timeout for slow operations
   - Rotate expired API keys

### Workflow 3: Verifying MCP Protocol Integrity

**Scenario**: Need to confirm stdio server isn't corrupting the MCP protocol stream.

**Steps:**

1. **Capture stdout output**:

   ```bash
   node dist/bin/oak-curriculum-mcp.js > stdout-test.txt 2>&1
   ```

2. **Verify all output is valid JSON-RPC**:

   ```bash
   # Should parse without errors
   cat stdout-test.txt | jq . > /dev/null && echo "✅ stdout is clean"
   ```

3. **Check for unexpected content**:

   ```bash
   # Look for non-JSON lines (should be empty)
   cat stdout-test.txt | grep -v '^{' | grep -v '^}' | head -10
   ```

4. **Verify file logging works**:

   ```bash
   # Should see recent logs
   tail -20 .logs/oak-curriculum-mcp/server.log | jq .
   ```

5. **If stdout is contaminated**:
   - Search code for `console.log()` → use logger instead
   - Check logger configuration → ensure file-only sink
   - Review dependencies → some may write to stdout
   - Check error handlers → ensure they use logger

### Workflow 4: Debugging Tool Input/Output Validation Issues

**Scenario**: Tool execution fails validation or returns unexpected data.

**Steps:**

1. **Find validation errors**:

   ```bash
   grep "validation" .logs/oak-curriculum-mcp/server.log | grep "error" | tail -10 | jq .
   ```

2. **Get correlation ID** and trace execution
3. **Extract validation error details**:

   ```json
   {
     "level": "error",
     "message": "Tool output validation failed",
     "context": {
       "payload": "{\"toolName\":\"...\",\"outputValidationFailed\":{\"message\":\"Missing required field: title\"}}",
       "toolName": "getLessonPlan"
     }
   }
   ```

4. **Review schema requirements**:
   - Check SDK tool definition for input schema
   - Review API response schema for output validation
   - Compare actual data against expected schema

5. **Common validation issues**:
   - Missing required fields
   - Wrong data types (string vs number)
   - Invalid enum values
   - Null where non-null expected

6. **Take action**:
   - Update tool input to match schema
   - Fix API response transformation
   - Update schema if API changed
   - Handle optional fields properly
