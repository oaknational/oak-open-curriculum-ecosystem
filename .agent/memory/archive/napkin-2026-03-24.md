### Architectural Finding: sdk-codegen boundary separation needed

sdk-codegen contains three distinct concerns that should be separated:
1. Generators (codegen.ts, zodgen.ts, bulkgen.ts, vocab-gen/) — pure functions
2. Generated API types (src/types/generated/api-schema/, zod/) — build output
3. Runtime code + bulk/vocab/synonym generators and output — different source

Proposal: sdk-codegen keeps API generators + generated API types. New workspace
for bulk/vocab/synonym generators + generated output + runtime code. This
eliminates within-workspace test dependencies and makes `pnpm clean && pnpm
check` work without turbo overrides.

### Lessons

- `--concurrency=2` was added as "fix: stabilise pre-commit turbo concurrency"
  but was actually masking missing dependency declarations. Throttling does not
  fix dependency bugs — it just makes the race condition less likely.
- Unit tests that call `existsSync` are doing IO. The testing strategy is
  clear: unit tests DO NOT trigger IO. Use the right tool for the job — tests
  prove behaviour, not file existence.
- Tests that import generated code to verify it are testing build artifacts.
  They belong downstream of the generation boundary, not in the generator
  workspace. This is an architectural boundary issue, not a turbo config issue.
- "Should any test depend on built code?" — tests that import from
  `src/types/generated/` create an implicit build-order dependency. Within the
  same workspace, this requires turbo overrides. Across workspaces, ^build
  handles it naturally. The workspace boundary IS the dependency declaration.
- The existing two-workspace separation (sdk-codegen + curriculum-sdk) is the
  right boundary shape, but some code is on the wrong side of it. Adding a
  third workspace for bulk/vocab/synonym concerns may be cleaner than moving
  everything to curriculum-sdk, because search-sdk also depends on this output.

### Key Architectural Finding: CLI-SDK Retriever Boundary

**The full retriever duplication landscape** (discovered via reviewer convergence):

| Scope | SDK retriever | CLI retriever | Status |
|-------|--------------|---------------|--------|
| Sequences | `buildSequenceRetriever` | ~~`createSequenceRetriever`~~ | Collapsed this session |
| Threads | `buildThreadRetriever` | ~~`createThreadRetriever`~~ | Collapsed this session |
| Lessons | `buildFourWayRetriever(..., 'lesson')` | `createLessonRetriever` | **Still duplicated** |
| Units | `buildFourWayRetriever(..., 'unit')` | `createUnitRetriever` | **Still duplicated** |

The CLI's `createLessonRetriever` and `createUnitRetriever` duplicate the SDK's
`buildFourWayRetriever`. All four architecture reviewers independently flagged
the sequence/thread duplication; the lesson/unit duplication is the same pattern.

**Principle**: The SDK is the capability layer; the CLI is the operational interface.
Retriever shape construction is a capability (domain contract), not an operation.
CLI should import retriever builders from SDK `/read`, keeping only filter
construction (app-specific) and request assembly (CLI-specific).

### Lessons

- Four parallel architecture reviewers converge on the same structural finding
  from different perspectives (DRY, coupling, ADR compliance, drift risk). This
  is strong signal — structural patterns found by 3+ reviewers are real.
- `isValidKeyStage` guard was silently skipping units before Wilma caught it.
  Defensive guards that skip instead of throwing are silent data loss in
  fail-fast pipelines.
- Code pattern documentation (`sdk-owned-retriever-delegation.md`) crystallises
  the mechanical fix so future sessions don't re-discover it.
