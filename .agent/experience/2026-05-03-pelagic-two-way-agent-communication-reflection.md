## What worked

### 1. The comms-event format carried substance reliably

The structured JSON event format (`event_id`, `created_at`, `author`,
`audience`, `in_response_to`, `title`, `body`) gave both agents a
consistent shape to read and write. No ambiguity about who said what
or to whom. The `in_response_to` field made threading explicit; Misty's
output-shape question and my reply formed a clear two-turn exchange.

### 2. Active-claims.json prevented duplicate writes

Misty's claim covering `smoke-tests/` workspace was visible to Pelagic
the moment it appeared in the file. Pelagic deferred any writes inside
that workspace (the planned harness/ subdirectory) until Misty's claim
closes. The claim's `intent` field carried enough semantics for
Pelagic to know the claim was read-only and that mutual-read coexistence
was safe.

### 3. The atomic-isolated task framing was load-bearing

Pelagic's Task M1 was deliberately:

- **Read-only** — no file writes; output is a single comms event
- **Bounded scope** — one specific deliverable (structured map of
  one directory tree)
- **Self-acceptance-criteria** — Pelagic could verify completion
  without re-walking the harness
- **No reviewer dependency** — research output, no review needed

This framing made the task graspable in one read. Misty accepted
without back-and-forth on scope. Atomic-isolated is the load-bearing
property; without it, parallelism creates more coordination cost than
it saves.

### 4. The shared-comms-log render gave a single chronological view

Auto-rendered `shared-comms-log.md` reading top-to-bottom showed the
full thread of events in time order. New sessions joining mid-stream
get the full chronology in one read. The render is regenerated by
tooling (not authored), so it stays current without coordination cost.

### 5. Owner-named explicit dispatch worked

Owner's *"please use the communication log to give them atomic,
isolated tasks"* was unambiguous about the protocol. No guessing about
whether to use comms log vs another channel. Single source of truth.

## What didn't work (or didn't work well)

### 1. The temporal-anomaly was confusing on first read

Misty's claim predated my task offer by 53 minutes. On first read this
looked like Misty had taken work I hadn't yet authorised. Without
context (the owner had given Misty the task brief separately), I
couldn't tell if Misty was running ahead of orchestration or in violation
of it. The protocol does not surface the *out-of-band brief* — it
assumes all instructions flow through comms log. In reality, owner
direction reached Misty before the comms-log was used.

**Cure shape**: when an agent acts on owner direction received outside
the comms log, the first comms event from that agent should explicitly
cite the out-of-band source (*"per owner direction at <approximate
time>, taking task X"*) so the orchestrator knows the brief came from
the owner, not from a missed comms event. This is a small protocol
amendment with high clarity payoff.

### 2. Path-overlap between Misty's read claim and Pelagic's planned writes

Misty's workspace claim (`smoke-tests/`) overlapped with Pelagic's
intended write area (`smoke-tests/harness/*`). Their claim was
read-only by intent but the schema doesn't carry a *read* vs *write*
flag. The path overlap meant Pelagic had to defer writes even though
the actual semantic conflict was zero.

**Cure shape**: extend the claim schema with `mode: 'read' | 'write'
| 'mutual-exclusive'` so non-conflicting modes can coexist on the
same path. Read claims would not block write claims on disjoint files
inside the same directory. This composes with the existing `area_kind`
field cleanly.

### 3. ETA discipline was weak

Misty quoted *"ETA: 30 min from claim open (08:02 UTC), so 08:32 UTC
reply"* but at 09:05 had not delivered. No heartbeat event was posted
explaining the overrun. The orchestrator (Pelagic) had no signal whether
Misty was stuck, productively going long, or gone. Heartbeat discipline
was named in the original task offer (*"heartbeat the claim if running
long"*) and acknowledged by Misty (*"will heartbeat the claim if I run
long"*) but not delivered.

**Cure shape**: heartbeat-or-die structural enforcement. If the claim
freshness exceeds the stated ETA + a grace window (e.g. 50%), the
orchestrator treats the claim as stale and either (a) reclaims and
reassigns, (b) opens an escalation event, or (c) blocks until owner
intervenes. Without structural enforcement, ETAs decay to wishes.
Pairs cleanly with the `freshness_seconds` field that already exists
on claims.

### 4. The output-shape coordination required an extra round

Misty raised a legitimate question (1200-word inline vs hybrid with
detail file) that Pelagic had to answer before Misty could complete.
That added a round-trip. The original task offer specified inline
1200 words but didn't anticipate the *what if it doesn't fit?* case.

**Cure shape**: task offers should include an *overflow protocol*
section — *"if the spec is too tight, do X; if X is also too tight,
escalate via Y."* This pre-empts the coordination round when the task
issuer can predict the failure mode. For tasks where overflow is hard
to predict, the orchestrator should explicitly authorise the safest
default upfront.

### 5. Bundle-leakage tension at commit time

When Pelagic prepared a session-open continuity commit, Misty's claim
state and acceptance event were live in the working tree as M and ??
files. Including them in Pelagic's commit was bundle-leakage (Misty's
substance attributed to Pelagic's commit). Excluding them left the
shared `active-claims.json` and `shared-comms-log.md` rendered files
in inconsistent states. Pelagic resolved by including them with
explicit attribution in the commit message, but the structural pattern
is unsatisfying.

**Cure shape**: make `shared-comms-log.md` a render artefact that is
regenerated deterministically from the events directory (it already is,
but the render-output's git status should be normalised between
agents). Either (a) auto-regenerate at commit time and have the commit
hook treat it as a generated file (analogous to `pnpm sdk-codegen`
output), or (b) `.gitignore` it and re-render on demand.

For `active-claims.json` (genuine shared mutable state), the cure is
finer-grained: when one agent commits while another's claim is open,
the commit should include a *snapshot* of the state with explicit
attribution that the included claim is another agent's. This is what
Pelagic did manually; the commit-skill should automate it.

### 6. Comms-event filenames as informal coordination

Filenames like `claude-f730bd-pelagic-misty-task-1-harness-recon.json`
encode metadata (platform, session prefix, agent name, task number,
topic) that the schema also carries inside the JSON. This duplication
risks drift if the filename and the body disagree. Filename also
becomes load-bearing for grep/glob discovery, but there's no formal
naming spec.

**Cure shape**: either (a) drop the filename's informational role and
rely on body fields for discovery (filenames become opaque UUIDs), or
(b) document the filename schema as part of the comms-log convention.
Option (a) is simpler; option (b) preserves the shell-grep ergonomics
of the current pattern.

## Smoother / easier / more reliable improvements

### Tooling-level

1. **`comms reply` CLI subcommand** that writes a new event with
   `in_response_to` populated, `audience` inverted from the source,
   and the source event's title prefixed with `Re:`. Removes manual
   JSON authoring at the most error-prone moment.
2. **`comms watch` blocking command** that tails the events directory
   and prints new events as they arrive. The orchestrator can leave
   it open in a separate pane; new events surface in seconds rather
   than at next manual check.
3. **`comms pending` query** — list events where I am in `audience`
   and no event with my `agent_name` carries `in_response_to: <event_id>`.
   That's "events awaiting my reply." Trivial to compute, valuable for
   not dropping requests.
4. **Heartbeat as a CLI ceremony** — `claims heartbeat <claim_id>
   --eta-revision "+30m"` posts a small event AND updates the claim's
   freshness window. Inertia of the manual JSON write is the reason
   the discipline degrades.

### Protocol-level

1. **Out-of-band brief acknowledgement** (cure for issue #1 above).
2. **Read/write claim mode** (cure for issue #2 above).
3. **Heartbeat-or-die enforcement** (cure for issue #3 above).
4. **Overflow protocol in task offers** (cure for issue #4 above).
5. **Render-artefact convention for `shared-comms-log.md`** (cure for
   issue #5 above).
6. **Filename convention spec** (cure for issue #6 above).

### Practice-level

1. **Atomic-isolated task taxonomy**. Today, "atomic isolated task"
   is a phrase. With a few worked examples (read-only research,
   pure-function authoring with TDD, type-only refactor at a single
   file scope, doc-only edit at a single file scope), the bar becomes
   teachable. The current Task M1 — read-only reconnaissance with a
   single comms-event output — is one canonical shape. A second
   shape: *implement one pure function with its unit test in one
   file pair, with a typed contract you can verify without running
   the integration*. A third: *audit one specific signal class
   (e.g. all `unknown` types in a workspace) and produce a structured
   findings list*.
2. **Coordinator vs executor role distinction**. In two-agent mode,
   one agent should typically take the coordinator role (designs the
   work, dispatches tasks, integrates results) and the other takes
   the executor role (accepts tasks, completes them, reports back).
   Today this is implicit; making it explicit in the thread record
   would reduce the *who's driving?* ambiguity.

## Scaling to more than two agents

The protocol generalises but stresses appear at specific points:

### What scales naturally

- **Append-only event format**. N agents post events; reader reads
  chronologically. Linear-in-events read cost.
- **Per-agent identity**. PDR-027's additive identity rule already
  supports arbitrary participation count. Each agent has a unique
  `session_id_prefix` and an additive thread-record row.
- **Claim-based coordination**. With per-area claims, N agents can
  work on disjoint areas with no coordination cost.

### What needs strengthening

- **Audience addressing**. With N>2 agents, `audience: ["agent-name"]`
  becomes ambiguous if names duplicate. Move to claim_ids or
  session_id_prefixes for unambiguous routing.
- **Topic threading**. With N agents posting on the same thread,
  finding the *current state of the discussion on topic X* across
  events from N parties becomes hard. Either (a) introduce explicit
  `topic_id` field with a typed taxonomy, or (b) build a CLI that
  groups events by `in_response_to` chains.
- **Conflict resolution**. With 2 agents, mutual-exclusive claims
  resolve via either-can-take-it. With N agents, there's no
  natural arbitration. Need either a deterministic priority order
  (by claim_id timestamp, by agent role, by orchestrator
  designation) or an explicit dispatcher role.
- **Render performance**. `shared-comms-log.md` linearly grows.
  With N agents posting at higher rates, the render becomes large.
  Either (a) paginate or shard by date, (b) introduce a `recent.md`
  that holds the last 7 days and an `archive/` for older events.
- **Trust assumptions**. Two agents in the same session-arc, owned
  by the same human, can trust each other implicitly. With N agents
  the trust chain is more diffuse — some may be running on different
  models, different instructions. Need explicit trust attestation
  in agent identity or claim metadata.

### What might break

- **Owner-as-orchestrator scaling cost**. If the human is the
  orchestrator for 3+ agents, the human becomes the bottleneck.
  The cure is a software orchestrator role that can dispatch
  atomic-isolated tasks based on a queue. This is a meaningful
  capability addition, not a small protocol amendment.
- **Heisenberg-style observation effects**. The act of one agent
  writing to `active-claims.json` perturbs other agents' reads.
  With 2 agents the conflict surface is small. With N agents,
  optimistic-locking or CRDT-style merge becomes necessary.
- **Vocabulary drift**. With 2 agents the vocabulary-trip-list and
  doctrine catalogue stay aligned. With N agents on different
  models or different platforms, vocabulary may drift, and the
  *"I think you mean X but you said Y"* coordination overhead
  grows. May need explicit shared-vocabulary refresh ceremonies.

## Net assessment

**For two agents on a focused arc, the protocol is workable.** The
comms log + active-claims + thread record triad gives enough structure
to coordinate without heavy tooling. The atomic-isolated task framing
is the load-bearing concept; if every dispatch satisfies it, the rest
falls into place.

**The biggest single improvement available** is heartbeat-or-die
enforcement on claim freshness. Without it, parallel work degrades
silently into orchestrator-blocked-on-missing-reply.

**The biggest scale risk** is owner-as-orchestrator cost. Two agents
fit one human's coordination budget; three agents probably do not. The
N-agent protocol needs a software orchestrator before it can scale.

**One emergent observation worth capturing**: when the orchestrator
(Pelagic) has substantial work of its own, dispatching to a peer is
itself work. The expected speedup from parallelism is offset by the
coordination cost. Atomic-isolated tasks help because they minimise
back-and-forth, but the floor cost of *frame the task, dispatch it,
read the reply, integrate it* is non-trivial. For tasks under ~30 min
of executor effort, the parallelism is probably net-negative. For
tasks over ~60 min where the orchestrator can do other work in the
meantime, parallelism is net-positive.

## Concrete amendments queued for graduation

These are immediate candidates for the pending-graduations register:

1. **Out-of-band brief acknowledgement** — when an agent acts on
   owner direction received outside the comms log, the first comms
   event from that agent must cite the out-of-band source.
2. **Read/write claim mode field** — extend `active-claims.json`
   schema with `mode: 'read' | 'write' | 'mutual-exclusive'`.
3. **Heartbeat-or-die enforcement** — claims past `claimed_at +
   freshness_seconds * 0.5` without a heartbeat event are stale.
4. **Overflow protocol in task offers** — task issuers must include
   *what to do if the spec is too tight*.
5. **`comms` CLI ergonomics improvements** — `reply`, `watch`,
   `pending`, `heartbeat`. The CLI ergonomics plan

   reflection adds operational specifics.
