# 🔍 Hybrid Search Matching Approaches in Elasticsearch Serverless

## Fuzzy Matching, RRF, Semantic Search (ELSER), and Synonyms

### (Technical Deep Dive for Implementation & Architecture)

This document explains how to combine **fuzzy lexical search**, **ELSER semantic search**, and **Reciprocal Rank Fusion (RRF)** in Elasticsearch Serverless. It also clarifies what each component does—and doesn’t—handle automatically.

---

# 1. Lexical (BM25) Search + Fuzziness

## ❌ Does Elasticsearch match “2” and “two”?

No.  
Elasticsearch **does not automatically normalize** numeric digits and their word equivalents.

- `"2"` tokenizes as a **digit**
- `"two"` tokenizes as a **word**
- They do **not** share stems
- They are **not** edit-distance neighbors  
  → `"2"` vs `"two"` → edit distance = 3 → fuzziness does NOT help

Therefore:

### ✔ Searching `"2"` will NOT match `"two"`

### ✔ Searching `"year 2"` will NOT match `"year two"`

### 🛠 You must explicitly configure this via synonyms

(See small synonym note at the end.)

---

## 1.1 Fuzziness _does_ help with common typos

Examples where fuzziness **does** help:

| Query      | Doc Term     | Match? | Why                        |
| ---------- | ------------ | ------ | -------------------------- |
| “frations” | “fractions”  | ✔      | edit distance 1            |
| “masss”    | “mass”       | ✔      | edit distance 1            |
| “fraction” | “fractional” | ✔/⚠️   | fuzzy + analyzer dependent |

But fuzziness does _not_ help with semantic transformations or number words.

### Canonical fuzzy configuration (lexical):

```json
{
  "multi_match": {
    "query": "user query",
    "fields": ["title^3", "description"],
    "fuzziness": "AUTO",
    "prefix_length": 1
  }
}
```

# 2. Semantic Search (ELSER)

ELSER works by encoding text into **sparse vectors** generated by the ML model. It captures **conceptual similarity**, such as:

- “topic” ≈ “subject”
- “teacher explanation” ≈ “instruction”
- “two apples” ≈ “2 apples” (sometimes)

BUT:

## ⚠️ ELSER DOES NOT GUARANTEE numeric normalization

It may understand:

- “two apples” ~ “2 apples”

But will probably **not** normalize:

- “lesson 2.1” ↔ “lesson two point one”
- “KS2” ↔ “Key Stage Two”
- “2nd” ↔ “second”

Semantic recall is _helpful_ but not deterministic.

You **cannot rely** on ELSER alone for number normalization.

---

# 3. Hybrid Search With RRF (Recommended)

RRF combines **lexical + semantic** results using document ranks, not scores.

Benefits:

- No manual score tuning needed
- High recall (semantic) + high precision (lexical)
- Robust to typos (via fuzziness)
- Robust to synonyms/meaning (via ELSER or synonym filters)

Example structure:

```json
{
  "retriever": {
    "rrf": {
      "retrievers": [
        {
          "standard": {
            "query": {
              "multi_match": {
                "query": "fractions year 2",
                "fields": ["title^3", "description"],
                "fuzziness": "AUTO"
              }
            }
          }
        },
        {
          "standard": {
            "query": {
              "sparse_vector": {
                "field": "description_semantic",
                "query": "fractions year two",
                "inference_id": "elser_model",
                "k": 50
              }
            }
          }
        }
      ],
      "rank_window_size": 50,
      "rank_constant": 60
    }
  }
}
```

Here’s what happens:

| Component                | Role                                                 |
| ------------------------ | ---------------------------------------------------- |
| **Lexical fuzzy match**  | Catches typos and exact-token matches; fast; precise |
| **ELSER semantic match** | Catches conceptual similarity                        |
| **RRF**                  | Merges the two signals into a final ranking          |

This is the _canonical_ hybrid search architecture in Elasticsearch 9.x / Serverless.

# 4. TypeScript SDK Example (Serverless + RRF + Fuzzy + ELSER)

```ts
import { Client } from '@elastic/elasticsearch';

const client = new Client({ node: 'https://<your-serverless-endpoint>' });

const res = await client.search({
  index: 'lessons',
  body: {
    retriever: {
      rrf: {
        retrievers: [
          {
            standard: {
              query: {
                multi_match: {
                  query: 'year 2 fractions',
                  fields: ['title^3', 'description'],
                  fuzziness: 'AUTO',
                  prefix_length: 1,
                },
              },
            },
          },
          {
            standard: {
              query: {
                sparse_vector: {
                  field: 'description_semantic',
                  query: 'year two fractions',
                  inference_id: 'elser_model',
                  k: 50,
                },
              },
            },
          },
        ],
        rank_window_size: 50,
        rank_constant: 60,
      },
    },
  },
});

console.log(res.hits.hits);
```

# 5. Should You Use Synonyms?

### Short answer: **YES**

For curriculum/education data like yours, synonyms are essential.

Elasticsearch does NOT automatically understand:

- `"2"` ↔ `"two"`
- `"KS2"` ↔ `"Key Stage 2"`
- `"Year 2"` ↔ `"Year Two"`
- `"grade three"` ↔ `"grade 3"`

Therefore, configure synonyms such as:

```
1, one
2, two
3, three
ks1, key stage 1
ks2, key stage 2
ks3, key stage 3
year 2, year two
```

Synonyms improve both:

- **lexical matching** (BM25)
- **semantic matching** (if applied at index time)
- **RRF effectiveness** (better lexical recall provides better fusion results)

# 7. Canonical Best Practices (Production Ready)

| Component                  | Purpose                                                        |
| -------------------------- | -------------------------------------------------------------- |
| **Fuzziness (AUTO)**       | Catch typos + spelling variation                               |
| **Synonyms**               | Handle number words, KS2 mappings, domain-specific equivalence |
| **ELSER semantic search**  | Understand conceptual meaning                                  |
| **RRF hybrid**             | Merge lexical + semantic into a superior ranking               |
| **Field boosts**           | Ensure important fields dominate lexical relevance             |
| **rank_window_size = 50+** | Ensures good recall in RRF                                     |

This approach yields the best practical search quality on Elasticsearch Serverless.

---

# 8. Final Takeaways

- **Elasticsearch does NOT automatically match numeric words ↔ digits.**
- **Fuzziness does NOT help with numbers → words.**
- **ELSER may help but is NOT reliable.**
- **Therefore you MUST use synonyms for numeric normalization.**
- **RRF hybrid search is the recommended production setup.**
- **The combined fuzzy + synonym + semantic + RRF pipeline gives the best possible recall and ranking.**
