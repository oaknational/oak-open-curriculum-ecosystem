Scaling ELSER Ingestion in Elasticsearch Serverless

Understanding Serverless Capacity and Scaling

Elastic’s Cloud Serverless is designed to autoscale its resources for indexing and ML inference workloads. There is no way to manually “resize” or add more power to a serverless instance – instead, the platform automatically provisions more ingest and ML compute capacity as needed ￼ ￼. In other words, you cannot directly configure CPU/RAM like a traditional cluster; the service monitors your load and scales behind the scenes. This means if your ELSER-based ingestion is saturating the default capacity, the serverless infrastructure should theoretically add resources (within limits) to handle it. However, in practice you may still encounter bottlenecks or transient limits if the ingestion spike is too fast for the autoscaling to react, or if you hit internal queues and quotas. In short, you can’t “make a serverless instance more powerful” on demand – you have to work within the autoscaling framework and optimize how you feed data into it.

One important aspect is that ELSER inference (semantic_text fields) consumes ML capacity. In serverless, the ML workload (for ELSER model inference) is separate from basic indexing. Elastic’s serverless platform allocates dedicated ML compute units for tasks like ELSER expansion ￼. The autoscaling will account for this, but heavy NLP processing can still become a throughput bottleneck compared to plain indexing. Elastic has recently introduced the Elastic Inference Service (EIS) with GPU acceleration for ELSER, which can dramatically boost throughput (up to 10× higher ingest throughput than CPU-based processing) ￼ ￼. On serverless projects, the default semantic_text setup already uses EIS (the managed inference endpoint) for ELSER, so you are likely leveraging that by default ￼ ￼. This is good for performance – EIS on GPUs is much faster – but it still has concurrency limits. There isn’t a toggle to further scale that on your end beyond what Elastic provisions, aside from ensuring you’re using the latest version (ELSER v2) and the GPU-backed inference. If your project was created earlier or not using EIS, consider switching to the managed .elser-2-elastic inference endpoint which uses EIS, as it streamlines scaling and performance. In summary, Elastic Cloud Serverless will auto-scale ingest and ELSER processing to an extent, but you cannot manually allocate a bigger node or increase instance size the way you might on a self-managed cluster.

ELSER Ingestion Throughput and Limits

It’s important to recognize that ELSER semantic indexing is computationally heavy, and there are known throughput limits. Elastic’s documentation and benchmarks indicate that with ELSER v2 (optimized model), a single 16 vCPU machine can index on the order of ~22–26 documents per second at maximum throughput ￼ ￼. This was observed using 16 parallel model allocations (one per CPU core) to fully saturate the hardware, yielding ~26 docs/sec in that setup ￼. In other words, ELSER is not built for extremely high indexing rates out-of-the-box – it trades speed for semantic relevance. If you try to ingest thousands of docs per second, ELSER will not keep up unless you can parallelize it across many cores or nodes. In your case, with ~12k documents (and 2 semantic_text fields each, so effectively ~24k inference operations) being ingested quickly via Bulk, it’s likely the ingestion rate exceeded what the ELSER model could process in real time. The symptom you saw – the ingestion queue filling up and documents getting dropped – is a classic sign of the model inference becoming a bottleneck.

Understanding the inference queue: Each model deployment (inference endpoint) in Elasticsearch has an internal request queue. By default it holds up to 1024 pending inference requests ￼. If that queue fills because requests are coming in faster than the model can handle, additional requests will be rejected (HTTP 429 errors), effectively dropping those documents from being indexed ￼. This is likely what happened in your case – ELSER couldn’t process fast enough, the queue hit its limit, and new bulk index requests started failing. Unfortunately, in serverless you might not see those 429 errors unless you inspect the Bulk API response or logs closely. It’s crucial to handle this scenario to avoid data loss.

Model allocations and throughput: ELSER can scale throughput by using multiple parallel model allocations (instances) rather than a single one. Elastic advises that to maximize ingest throughput, you should use as many model allocations as there are CPU cores (each allocation single-threaded) ￼ ￼. For example, on a 32-core machine, running 31 or 32 allocations with 1 thread each is ideal for ingest throughput ￼. The default deployment, however, often uses only 1 allocation and 1 thread (a conservative default) ￼. That default effectively uses just one core, significantly limiting throughput. In a self-managed cluster you could increase number_of_allocations (and ensure each has num_threads:1 for ingest) to use all available cores ￼ ￼. On serverless/EIS, you don’t directly set this on the default endpoint – but you can create a custom inference endpoint with specific settings if needed. For instance, using the Create Inference API you could deploy the ELSER model with num_allocations > 1 (to allow more parallel inference) and num_threads: 1 (optimized for ingest) ￼ ￼. Increasing allocations generally increases throughput linearly (until you saturate CPU/GPU), while using more threads speeds up single inference latency (better for search) ￼. For ingestion, use many single-threaded allocations rather than a few multi-threaded ones ￼.

In Elastic’s cloud, if you were using a hosted (non-serverless) deployment, the advice would be to add more ML nodes or increase node size to get more CPU cores, deploy the ELSER model there, and bump up the allocations to scale out throughput ￼ ￼. Each additional ML node effectively adds another copy of the model that can double the docs/sec throughput (e.g. two 16-vCPU nodes could handle ~52 docs/sec instead of ~26) ￼ ￼. On serverless, you cannot manually add ML nodes, but the autoscaling should provision more behind the scenes if sustained load demands it. The caveat is that autoscaling may not instantaneously allocate 32 CPUs just because you sent a big batch – it might react gradually. There may also be upper limits per project or throttle in how fast it scales. If you consistently need higher throughput, consider whether Elastic Cloud (traditional) with explicit ML node scaling might serve better for the bulk indexing phase, after which you could switch back to serverless. This is a heavy-handed approach, but it gives you manual control to throw hardware at the problem during ingestion.

Optimizing Ingestion with ELSER (Configuration and Design)

Since direct vertical scaling of serverless is not an option, the solution is to optimize your ingestion pipeline and ELSER processing. Here are key strategies and settings:
• Use a dedicated inference endpoint with optimized settings: You can override the default by creating your own inference endpoint for ELSER (using the Elastic Inference API) and pointing your semantic_text fields to it via an inference_id in the mapping ￼ ￼. This allows you to tune parameters. Set num_threads: 1 for throughput-oriented mode (as noted, multi-threading a single inference is only useful for search latency) ￼. Increase num_allocations to deploy multiple instances of the model – each acts like a worker that can process one document at a time in parallel ￼. For example, if the default .elser-2-elastic endpoint is single-instance, creating an endpoint with, say, 4 allocations (and 1 thread each) could roughly quadruple your throughput capacity (assuming enough ML compute is available). Elastic also offers adaptive allocations, where the system will automatically scale the number of model instances up or down based on load ￼ ￼. Enabling this (via adaptive_allocations.enabled: true and setting reasonable min/max limits) can help the model deployment dynamically respond to surges in documents. Essentially, it’s an auto-scaling inside the model: when the load is high, it spins up another allocation (up to a max you allow) ￼ ￼. This could be very useful in serverless, though it’s a newer feature – you’d configure it through the API since the UI might not expose it. The bottom line is increasing the parallelism of the ELSER model is the way to handle higher ingest rates, but it may require custom configuration since the out-of-box default might be one single process.
• Separate ingestion and search workloads: If you are simultaneously querying the data (semantic search queries) while ingesting, they could contend for the same model resources. In such cases, a best practice is to deploy ELSER twice – one instance for indexing, and one for querying ￼. By giving your semantic_text fields an inference_id that points to a model dedicated to ingestion, and using a different model (or deployment) for search queries, you isolate the workloads. This prevents heavy indexing from flooding the model’s queue and degrading search latency ￼. On serverless/EIS, this could mean having two inference endpoints (both using ELSER model under the hood) – one receives all the ingest pipeline calls, the other is used at query time for, say, semantic reranking if you do that. This strategy ensures each endpoint has its own queue and throughput budget. It’s a more advanced setup, but worth considering if you expect ongoing high-volume indexing alongside query traffic.
• Optimize document and field sizes: Large documents or many fields per document significantly slow down ELSER processing. ELSER will only consider the first 512 tokens of each field by design ￼ ￼, but very large text fields still incur parsing and tokenization cost and may not improve search relevance beyond a point. If you have extremely large text (like long articles), consider chunking the content into smaller pieces (e.g. store multiple chunks in an array of semantic_text, or as separate docs) so that each piece is under ~512 tokens ￼. This can improve relevance and keep each inference task bounded in cost. Similarly, be mindful that you have 2 semantic_text fields per document – that doubles the workload per doc. If one of those fields is less important, or could be combined with the other, it might reduce workload. In most cases you can’t avoid it, but it’s something to evaluate. Also, clean the input text to remove any unnecessary content before ingestion (HTML tags, code, etc.) so that the model isn’t wasting effort on tokens that don’t help search ￼. By streamlining the text content, you slightly reduce the token count and processing time per document.
• Ensure ELSER is up-to-date and warmed up: Use the latest version of the ELSER model (ELSER v2 is much faster and more relevant than v1). In Elastic Cloud this should be the default by now. Also, when doing a large batch ingest, it helps to have the model already loaded and running. In serverless, the first request will trigger the model to load if it wasn’t in memory. You can index a few docs first and check that the inference endpoint is active. Elastic recommends setting a minimum allocation of 1 (rather than letting it scale to 0 when idle) if consistent responsiveness is needed ￼ – this avoids cold-start latency. On serverless/EIS the model is likely kept ready, but it’s something to keep in mind if there were periods of no activity. Essentially, treat the ELSER model as a component that might need some warm-up; don’t unleash thousands of docs in one go from cold start.

Ensuring Reliable Ingestion (Queuing and Throttling)

Even with optimizations, you should implement back-pressure and queuing on the client side to protect against data loss. Since we know the model can only ingest so fast, the goal is to send data at a rate that the system can sustain, or otherwise queue it externally until Elasticsearch can catch up. Here are some best practices:
• Throttle your bulk ingestion rate: Avoid blasting all 12k documents in one giant bulk or in a tight loop without delays. Instead, break the indexing into manageable batch sizes and insert deliberate pauses or rate limits. Monitor the indexing throughput (docs indexed per second) and adjust your feed rate to roughly match what ELSER can handle (for example, if it seems to process ~20 docs/sec, don’t send 200 docs/sec). If you are using the Reindex API or a similar mechanism, take advantage of its requests_per_second parameter to throttle the operation ￼. You can also use the \_rethrottle endpoint to dynamically adjust if you started too fast ￼. The principle is to smooth out the ingest flow rather than spiking. This gives the autoscaling a chance to keep up and prevents overwhelming the model’s queue.
• Handle 429/503 errors with retries: Make sure your ingestion client checks for failed bulk actions. If Elasticsearch responds with errors for some documents (e.g. a 429 Too Many Requests or a pipeline timeout), those docs should be retried after a delay. Implement an exponential backoff retry for rejected documents. This is critical – by default a bulk request dropping documents means data loss unless you catch it. The Elasticsearch client libraries typically return which items failed; you can requeue them. By continuously adjusting (backing off when errors occur), you create a feedback loop that naturally throttles ingestion to what the cluster can accept.
• Use an external queue or buffer: If possible, decouple your data source from direct indexing by using a message queue (like Kafka, AWS SQS, etc.) or a buffering component (Logstash, Beats with persistent queue, etc.). For instance, you could publish all documents to a queue and have a consumer process that reads from the queue and indexes into Elasticsearch at a controlled rate. This way, if Elasticsearch is slow or temporarily scales down, the data simply accumulates in the queue and isn’t lost – you have backpressure all the way to the source. Logstash with the Elasticsearch output is an option; it has a persistent disk queue and can catch failures. Even a simple script that reads input and writes to ES could be modified to sleep or wait when encountering throughput issues. The key is don’t rely on Elasticsearch’s internal ingest queue as the only buffer, because as we saw it has a fixed capacity (1024 by default) ￼. It’s better to control the feed externally and let ES ingest pipeline work steadily.
• Adjust bulk request size: Sometimes using slightly smaller bulk batches can improve throughput and reduce contention. If, for example, you were sending huge bulk payloads of say 5k documents in one request, consider using batches of a few hundred at a time. Smaller batches complete faster and free up resources, and they also distribute the work in time. There’s a balance to find – too small and you don’t get the efficiency of bulk; too large and you overload the pipeline. Experiment with batch sizes (e.g. 100, 500, 1000 docs per bulk) to see which yields stable performance.
• Monitor ingestion metrics: On Elastic Cloud (serverless) you may have monitoring graphs or logs indicating ingest pipeline throughput, queue depth, or ML node utilization. Keep an eye on those. In particular, \_nodes/stats on a regular cluster shows ingest.current and ingest.queue lengths for pipelines ￼ – in serverless, these details are abstracted, but any exposed metrics on the dashboard could hint if you’re saturating something (CPU, etc.). If you see the throughput leveling off or queue metrics maxed out, that’s a sign to slow down input. Also watch your ELSER token usage if on a paid plan – the pricing is per million tokens ￼, and extremely fast ingestion can run up costs or hit limits.

By implementing the above, you ensure that no documents get silently dropped. The goal is a reliable pipeline: either the data is indexed successfully, or it’s held somewhere (or retried) until it can be. It might take a bit longer to ingest all 12k documents, but it’s better to succeed slowly than to lose data quickly.

Conclusion and Best Practices Recap

To summarize, scaling ELSER ingestion on Elasticsearch Serverless requires architectural adjustments rather than a simple scale-up knob. You cannot directly give the serverless instance more horsepower; instead you must optimize how you use it and how you feed data into it:
• Leverage parallelism in ELSER: Use multiple model allocations (inference processes) with single threads to increase throughput ￼ ￼. In serverless, this may involve creating a custom inference endpoint or relying on Elastic’s EIS autoscaling to add capacity. Ensure you’re on the faster ELSER v2 model (and using GPU inference by default on serverless for maximum throughput ￼).
• Avoid pipeline overload: Don’t fire-hose all data at once. Throttle your ingestion rate to what the system can handle, and use bulk sizes and concurrency levels that balance speed with stability. Implement retries on failures and consider an external queue to buffer load ￼.
• Adjust Elasticsearch configuration if possible: If you have control (in a non-serverless scenario), raise the model deployment’s number_of_allocations to use all available cores and consider increasing the queue_capacity from the default 1024 to a higher value to buffer more requests (up to 1,000,000) ￼. On serverless, you can’t set cluster settings, but using separate inference endpoints for ingest vs. search can effectively give you more total capacity ￼.
• Design for reliability: Ensure no document is dropped by handling backpressure properly. It’s better to temporarily slow down ingestion or queue documents than to lose them. Use the tools provided (e.g., reindex throttling, persistent queues, autoscaling) to achieve a smooth pipeline.
• Consult Elastic’s guidance and limits: Elastic’s documentation on ELSER and semantic_text contains performance tips and notes on limitations ￼ ￼. For example, autoscaling will try to provide bigger ML nodes if needed, but there is an upper bound and diminishing returns after a point. Real-world throughput will depend on your document sizes and number of fields, so run tests to find your sustainable ingest rate. If you find that serverless is persistently not keeping up even after optimizations, engage with Elastic support – there could be project-level limits or something misconfigured.

In conclusion, making the ingestion pipeline reliable and performant is about working with the Elastic platform’s scaling features and not overrunning them. By tuning the ELSER inference setup (using more parallelism, possibly via custom endpoints) and by introducing proper flow control (queuing and throttling) on your ingestion process, you should be able to successfully ingest your large dataset without drops. While you may not achieve blazing speed due to the nature of semantic encoding, you can reach a stable throughput that gets all documents indexed with ELSER processing intact. This ensures your semantic search use case is powered by all your data, with no documents left behind due to ingest overload. Good luck, and happy indexing!
