## Metacognitive Pass

The inherited shape was "learn from the external skills repo and apply the
benefits". A literal plan-import shape would be wrong: the Practice is the
substrate, and skills are only one operational expression of it. The plan
therefore routes learning through a Practice-fit pipeline:

1. preserve the source study as evidence;
2. extract candidate concepts;
3. review each candidate against existing Practice surfaces;
4. promote only the smallest durable Practice-native changes;
5. defer or reject ideas that do not survive the fit review.

The bridge from action to impact is not adding more skills. The bridge is making
useful external substrate concepts become operational where Oak has a real
failure mode: a rule, directive, workflow, protocol, skill, state surface, or
validator that changes agent behaviour.

## Problem And Intent

The external substrate study identified several useful concepts: feedback-loop
first debugging, prototype expiry, skill dependency classes, glossary-like
alignment caches, agent briefs, manifest validation, referential handoff, and
negative-decision memory. They are promising, but they currently sit as research
findings rather than routed Practice work.

Without a plan, those findings can decay in two ways:

- as passive inspiration that never changes behaviour;
- as over-eager skill copying that bypasses the Practice's stronger substrate.

The intent of this plan is to turn the research into a bounded
research-to-Practice adoption path. The path keeps Oak's existing doctrine,
claims, comms, memory, gates, and agent-tools model intact while extracting the
small named flows that genuinely improve the Practice.

## End Goal, Mechanism, And Means

### End Goal

Oak gains the useful behavioural benefits from the external substrate study
without importing unreviewed source-specific assumptions, weakening existing
Practice doctrine, or creating duplicate operational surfaces.

### Mechanism

Each candidate concept is tested against the Practice substrate before adoption:

- Does it address a repeated Oak failure mode?
- Is Oak already solving it elsewhere?
- Is the current Oak surface too heavy, hidden, passive, or underspecified?
- Which operational expression is the right home: doctrine, rule, skill,
  workflow, protocol, agent-tool state, validator, or research note?
- What deterministic evidence would prove the change altered behaviour?

This keeps adoption mechanism-led rather than skill-led.

### Means

The strategic means are:

1. create a candidate register from the substrate study;
2. run a Practice-fit review for each candidate;
3. promote the first small behaviour-changing slices;
4. leave rejected, deferred, and observation-only ideas in explicit homes;
5. validate any promoted work through focused checks and the relevant
   consolidation loop.

## Domain Boundaries

### In Scope

- Candidate extraction from the substrate study.
- Practice-fit review against existing rules, directives, skills, protocols,
  memory, plans, and tools.

- A first prototype-expiry doctrine note if no existing home already captures

  it is not already covered by current lifecycle triggers.

### Out Of Scope

- Copying external skills into Oak.
- Adding source-specific names, URLs, or identity-bearing references to tracked

- Replacing existing Oak TDD, claims, comms, handoff, planning, or
  consolidation doctrine.
- Creating new agent-tools validators before the behaviour and data model are
  reviewed.
- Rewriting the full Practice substrate in one pass.
- Treating the ignored local reference snapshot as a portable dependency.

## Candidate Register Shape

**Decision**: the candidate register lives at
`.agent/research/agentic-engineering/operating-model-and-platforms/external-skills-substrate-candidate-register.md`.
It is a research-to-Practice routing artefact, not an executable plan and not a
second source of truth for doctrine.

The first promoted executable slice should create or update a candidate register
with this minimum shape:

| Field | Purpose |
| --- | --- |
| Candidate | Short name, e.g. `feedback-loop-first-diagnosis` |
| Source insight | Link to the tracked study section, not source identity |
| Oak failure mode | Concrete behaviour this would improve |
| Existing home | Current directive/rule/skill/tool that may already cover it |
| Fit verdict | `promote`, `defer`, `reject`, or `observe` |
| Natural home | Plan, rule, directive, skill, protocol, tool, memory, or note |
| Proof signal | How a future agent can tell behaviour changed |

The register should be small. Candidate count is not a success measure.

## Candidate Concepts To Review

### C1. Feedback-Loop-First Diagnosis

**Strategic routing verdict**: promote to the first Practice-fit review.

**Hypothesis**: Oak debugging sessions would improve if a named workflow required
making the failure local, fast, and repeatable before attempting fixes.

**Likely home if promoted**: a small skill or workflow that references
testing-strategy and TDD-as-design rather than restating them.

**Fit question**: Does existing TDD doctrine already cover debugging sessions
well enough, or does a named diagnosis mode reduce missed-signal repairs?

### C2. Prototype Expiry

**Strategic routing verdict**: promote to the first Practice-fit review.

**Hypothesis**: Research and spike work needs an explicit expiry rule: a
prototype answers a question, then is deleted, absorbed, or promoted into a
durable artefact.

**Likely home if promoted**: a short rule or planning component for research and
spike work; not a broad principle unless the fit review finds repeated
cross-lane pressure.

**Fit question**: Is this already covered by plan lifecycle and consolidation,
or does prototype work need its own phrase because residue becomes architecture?

### C3. Skill Dependency Classes

**Strategic routing verdict**: defer until after C1 and C2. This is promising
but structural; it should not be bundled with the first behaviour slice.

**Hypothesis**: Skills and workflows would be easier to invoke correctly if they
declared hard and soft dependencies: repo grounding, claims, comms, plan
authority, specialist review, gates, or external configuration.

**Likely home**: skill metadata guidance plus plan/lifecycle component updates.

**Fit question**: Does this reduce both under-grounding and over-ceremony, or
would it duplicate existing start-right and lifecycle-trigger doctrine?

### C4. Glossary-Like Alignment Caches

**Strategic routing verdict**: observe. Promote only if a live lane shows
repeated term-negotiation friction that memory, plans, and ADRs are not solving.

**Hypothesis**: Some domains need a compact term glossary separate from plans,
ADRs, and memory so agents and the owner stop re-negotiating local vocabulary.

**Likely home**: research-lane guidance or a small optional template.

**Fit question**: Which lanes have enough term friction to justify a local
glossary without creating another passive document?

### C5. Agent Brief Object

**Strategic routing verdict**: defer. Revisit only when a multi-agent work
packet needs something smaller than an executable plan and stronger than a
handoff paragraph.

**Hypothesis**: A compact work-packet shape could improve multi-agent handoffs:
task, context, constraints, success signals, test expectations, and warnings.

**Likely home**: plan component or collaboration-state work-packet guidance.

**Fit question**: Does this complement executable plans, or would it create a
shadow plan with weaker authority?

### C6. Referential Handoff Compression

**Strategic routing verdict**: observe and opportunistically apply. This is
already close to Oak's handoff doctrine; only amend the handoff surface if a
future handoff pass shows the phrase or examples would prevent real drift.

**Hypothesis**: Handoffs should more explicitly prefer pointers to authoritative
artefacts over restating plan, ADR, commit, and thread-record content.

**Likely home**: session-handoff skill or thread-record guidance.

**Fit question**: Is the existing handoff flow already doing this, and if so is
the missing piece only a phrase and examples?

### C7. Inventory Validation

**Strategic routing verdict**: defer until there is live Oak inventory drift or
a separate skill/adapter inventory model lands.

**Hypothesis**: Skill indexes, plugin metadata, adapter files, and installable
surfaces need validation so declared availability cannot drift from actual
availability.

**Likely home**: portability or skills validator, only after the inventory model
is explicit.

**Fit question**: Is there current drift in Oak's skill and adapter inventory,
or is this a future guard with no present failure mode?

### C8. Negative-Decision Memory

**Strategic routing verdict**: defer. Promote only with evidence that rejected
or deferred routes are being rediscovered despite existing plans, ADRs, and
memory.

**Hypothesis**: Repeated "not this route yet" decisions need a small searchable
home with revisit conditions, so rejected lanes do not get rediscovered or
harden into unexamined constraints.

**Likely home**: memory guidance or pending-graduations adjacent note.

**Fit question**: Are current plans and ADR alternatives enough, or do repeated
defer/reject decisions disappear into logs and reappear as confusion?

## Strategic Acceptance Criteria

- **AC1 Candidate routing**: each candidate from this plan has a fit verdict,
  natural home, and proof signal before implementation work starts.
- **AC2 No duplicate authority**: promoted work references the authoritative
  Practice home and does not create shadow doctrine.
- **AC3 Small first landings**: first executable plan contains no more than two
  behaviour-changing Practice slices plus any necessary register/index updates.
- **AC4 Source-neutral tracking**: tracked artefacts keep source identity out of
  repo-facing docs and rely on the neutral research report as citation surface.
- **AC5 Operational expression**: every promoted candidate names whether it
  becomes doctrine, rule, skill, workflow, protocol, state, tool, or memory.
- **AC6 Evidence before completion language**: no slice is called complete until
  its proof signal has been checked by command, review, or explicit observation.

## Success Signals

The plan has succeeded when:

- the candidate register exists and prevents passive-inspiration drift;
- one or two high-confidence candidates are promoted into a current executable
  plan with scoped acceptance criteria;
- low-confidence ideas are explicitly deferred, rejected, or observed;
- any first implementation leaves a clearer invocation path for future agents;
- consolidation can mine the result into a stable Practice home without
  re-reading the external source.

## Risks And Mitigations

| Risk | Why it matters | Mitigation |
| --- | --- | --- |
| Skill-import bias | External skill names may look more ready than they are | Require Practice-fit verdict and natural home before any implementation |
| Passive-document drift | The report remains interesting but unused | Candidate register has proof signals and promotion triggers |
| Duplicate doctrine | New notes restate existing TDD, handoff, or planning rules | Existing-home check is blocking |
| Over-tooling | Validator work starts before the data model is proven | Keep validator as design-only until drift evidence exists |
| Privacy regression | Source identity leaks into tracked docs or comms | Use neutral citation surface and private ignored reference only |
| Scope sprawl | Eight candidates turn into one oversized execution plan | Promote at most two behaviour-changing slices at a time |
