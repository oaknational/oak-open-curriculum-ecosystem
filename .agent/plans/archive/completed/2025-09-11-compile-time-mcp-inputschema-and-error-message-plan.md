## Context and Problem Statement

Tools generated by the Oak Curriculum SDK via `pnpm type-gen` are consumed by the MCP server and clients. Two issues were identified:

- inputSchema published by the server is lossy and incomplete (types collapsed to string/number, no enum, no arrays, no booleans), causing clients (e.g. MCP Inspector) to not render correct parameter UIs or “not list” the parameters effectively.
- Parameter validation errors surfaced to clients show a placeholder message without schema details. The generator emits `getValidRequestParamsDescription()` returning only “Invalid request parameters. Please match the following schema:” with no actual schema content.

Root causes were confirmed in code review:

- SDK generator currently does not emit a JSON Schema for input; server reconstructs a simplified schema (`apps/oak-curriculum-mcp-stdio/src/tools/runtime/index.ts`).
- SDK generator’s error message emitter in `emit-schema.ts` intentionally returns a placeholder string.

## Goals

- Generate accurate JSON Schema for tool inputs at compile time in the SDK typegen and export it with each tool.
- Replace server-side schema reconstruction with a pass-through of the generated inputSchema (no compatibility layers).
- Generate helpful, structured, compile-time error descriptions that include the schema and required fields summary.
- Execute all work with strict unit-test-first TDD and keep all types/validators/schemas constructed at compile time (during `pnpm type-gen`).

## Non-goals

- Changing OpenAPI source or the API itself.
- Adding runtime schema builders or validators beyond what is generated at typegen time.
- Introducing compatibility layers. We replace old approaches directly.

## Constraints and Principles (aligned with references)

- TDD always (principles.md, Testing Strategy). Unit tests first, then implementation, then refactor.
- TypeScript best practice: no `any`, no `as`, prefer type guards and literal-preserving `as const` (TypeScript Practice).
- Single source of truth for types (principles.md, TypeScript Practice). Schema and enums flow from OpenAPI to generated constants.
- Fail fast with clear errors (Development Practice). Preserve cause chains, no silent failures.
- Quality gates order: type-gen → format → type-check → lint → test → build (Development Practice, principles.md).
- No compatibility layers; replace the old server-side mapping with SDK-provided schema (principles.md).

## High-level Approach

1. Input schema first (compile-time in SDK): extend the SDK typegen to emit a full JSON Schema for each tool’s parameters directly into the generated tool module, typed with literal types and exported.
2. Server schema pass-through: update the server to consume the tool’s exported inputSchema with no transformations.
3. Error message generation (compile-time in SDK): replace the placeholder error description with a pre-rendered, formatted, and concise description that includes a compact JSON representation of the schema and a required fields list.

## Detailed Work Streams

### A. Compile-time inputSchema generation (PRIORITY 1)

- Source of truth: OpenAPI parameters -> existing generator metadata (pathParams/queryParams) in `packages/sdks/oak-curriculum-sdk/scripts/typegen/mcp-tools/`.
- Extend the generator to emit a JSON Schema object per tool:
  - Type mapping:
    - string → `{ type: 'string' }`
    - number → `{ type: 'number' }`
    - boolean → `{ type: 'boolean' }`
    - string[] → `{ type: 'array', items: { type: 'string' } }`
    - number[] → `{ type: 'array', items: { type: 'number' } }`
    - boolean[] → `{ type: 'array', items: { type: 'boolean' } }`
  - Enumerations: when `allowedValues` present, emit `enum: [...]` with literal preservation.
  - Required: emit `required: [...]` from `required: true` flags.
  - additionalProperties: set to `false`.
  - Structure: `{ type: 'object', properties: { ... }, required: [...], additionalProperties: false }`.
- Emission location:
  - In each generated tool file (e.g. `src/types/generated/api-schema/mcp-tools/tools/<tool>.ts`) export `const inputSchema = { ... } as const`.
  - In the tool’s exported object, include `inputSchema` so consumers can reference `MCP_TOOLS[name].inputSchema`.
- Server updates (same change set, no compat layer):
  - Remove `apps/oak-curriculum-mcp-stdio/src/tools/runtime/index.ts` schema reconstruction; instead, list tools with `tool.inputSchema`.
  - Remove backwards-compatibility exports:
    - In `apps/oak-curriculum-mcp-stdio/src/tools/runtime/index.ts`, delete `export const tools = getMcpTools();` and any related compatibility comments.
    - In `apps/oak-curriculum-mcp-stdio/src/tools/index.ts`, delete `export type McpOrgan = McpToolsModule;` and `export function createMcpOrgan(...)`.
  - Ensure all imports/usages across the app reference only `getMcpTools` and `createMcpToolsModule`.
  - Documentation comments:
    - Inline JSDoc on the generated `inputSchema` constant and properties to satisfy typedoc.

### B. Compile-time error description generation (PRIORITY 2)

- Replace `getValidRequestParamsDescription()` placeholder in generator with a compile-time rendered description that includes:
  - Short heading.
  - A compact pretty-printed JSON block of the same `inputSchema` object.
  - A concise list of required fields.
- Emission:
  - Embed the exact rendered text as a `const` function result in each generated tool module to avoid runtime computation.
- Note on size: keep description small (single-level pretty print) to avoid noise.

## Testing Strategy (Strict Unit-test TDD)

- Unit tests (Vitest), written first, colocated with code they test.

A. SDK generator unit tests (in `packages/sdks/oak-curriculum-sdk/scripts/typegen/mcp-tools/`)

- `parts/param-utils.unit.test.ts`:
  - Prove primitive mapping for string/number/boolean and array variants.
- `parts/emit-params.unit.test.ts`:
  - Prove `pathParams/queryParams` metadata generation reflects required and enum values.
- `parts/emit-input-schema.unit.test.ts` (new):
  - Given sample operations, prove emitted schema includes:
    - Correct `type` for each property including arrays/booleans.
    - `enum` arrays when applicable.
    - `required` contains all and only required fields.
    - `additionalProperties: false`.
- `parts/emit-schema.unit.test.ts` (augment):
  - Prove the generated error description contains the pretty-printed JSON schema and required list.
- End-to-end typegen unit test (pure, in-process):
  - Generate tools for a minimal synthetic OpenAPI fragment and assert the produced tool module strings contain the expected `inputSchema` and error description snippets.

B. Server unit tests (in `apps/oak-curriculum-mcp-stdio/src/tools/`)

- `runtime/tools-list.unit.test.ts` (new):
  - Prove `getMcpTools()` returns the `inputSchema` exactly as emitted by SDK (no transformation).

C. Existing integration and E2E tests

- Update assertions in `e2e-tests/mcp-protocol.e2e.test.ts` to verify tools/list include:
  - Correct `required` and `enum` arrays for representative tools.
- Add a negative call check that the error message now includes the compact schema block.

Naming conventions per Rules:

- Unit tests: `*.unit.test.ts`
- Integration tests: `*.integration.test.ts`
- E2E tests: `*.e2e.test.ts`

## Step-by-step Plan (TDD loops)

1. Guardrail tests (RED)

- Write unit tests for new inputSchema emission and error message emission (generator parts).
- Write server unit test to require pass-through `inputSchema`.

2. Implement inputSchema emission (GREEN)

- Add a new emitter (e.g. `emit-input-schema.ts`) or extend `emit-schema.ts` to also return `inputSchema` text.
- Inject `inputSchema` into each generated tool file and into the tool export.
- Update `index.ts` emission if needed to preserve types.
- Update server tool listing to consume `tool.inputSchema`.

3. Refactor and clean (REFACTOR)

- Remove the server’s `generateInputSchema` and related mapping code entirely (no compatibility layers).
- Remove the compatibility exports and aliases in tools modules:
  - `runtime/index.ts`: drop `export const tools = getMcpTools()`.
  - `tools/index.ts`: drop `McpOrgan` type alias and `createMcpOrgan` function.
- Ensure names and boundaries are clear; add JSDoc on new/changed functions.

4. Implement error description emission (RED → GREEN → REFACTOR)

- Extend generator to render compact schema JSON and required list in `getValidRequestParamsDescription()`.
- Ensure TypeScript literal preservation and no assertions.

5. Quality gates and docs

- Run: `pnpm type-gen` → `pnpm format` → `pnpm type-check` → `pnpm lint` → `pnpm test` → `pnpm build`.
- Ensure typedoc can process new JSDoc comments.

## Acceptance Criteria

- Tools listed by MCP server include an `inputSchema` with:
  - Correct types for string, number, boolean, and arrays.
  - `enum` arrays where applicable.
  - Accurate `required`.
  - `additionalProperties: false`.
- At least three representative tools validated by unit tests and E2E to include enums and required fields.
- Parameter validation error messages include a compact JSON schema block and required list.
- No server-side schema reconstruction remains; server passes through SDK `inputSchema`.
- No backwards compatibility exports remain in `apps/oak-curriculum-mcp-stdio/src/tools/index.ts` and `apps/oak-curriculum-mcp-stdio/src/tools/runtime/index.ts`; all consumers use `createMcpToolsModule` and `getMcpTools`.
- No `any`, no `as` type assertions; literal types preserved with `as const` where appropriate.
- All tests written first and pass; E2E sanity checks pass.

## Risks and Mitigations

- **Risk:** Duplicate schema logic if both emitters and server mapping coexist.
  - **Mitigation:** Replace server mapping in the same PR, remove old code.
- **Risk:** Large error messages.
  - **Mitigation:** Compact JSON, shallow print, keep within reasonable length.
- **Risk:** Enum/array/boolean edge cases across all endpoints.
  - **Mitigation:** Unit tests cover representative cases; synthetic OpenAPI fragments used in tests.
